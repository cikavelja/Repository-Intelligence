# Repository Intelligence

A local-first proof-of-concept that indexes large software repositories, detects file changes, performs semantic-style code search, and recommends the most relevant files for a given issue — giving AI coding agents a focused working set before they make changes.

---

## Problem

AI coding agents struggle on large repositories because they cannot reliably understand the entire codebase at once. They may:

- Modify the wrong files
- Miss architectural boundaries
- Ignore existing conventions
- Solve only symptoms, not root causes
- Make unrelated changes that break other functionality
- Lose context in very large solutions

---

## Solution

This project builds a **Repository Intelligence Index** that:

1. Scans a source-code repository
2. Parses and indexes files into searchable chunks (using Roslyn for C#)
3. Stores metadata in SQLite
4. Detects added, updated, deleted, and unchanged files on each run
5. Performs hybrid semantic-style + keyword search over indexed code
6. Recommends which files are most relevant for a given issue
7. Explains why each file is relevant, lists related symbols, and suggests what to check
8. Generates a focused Copilot Agent prompt containing only the relevant files

## How Indexing Works

1. `LocalFileScanner` walks the repository directory tree
2. `GitIgnoreStylePathFilter` rejects `bin`, `obj`, `node_modules`, `.git`, etc.
3. For each supported file (`.cs`, `.md`, `.json`, `.ts`, `.sql`, `.yml`, …):
   - Content is read and SHA256-hashed
   - `.cs` files are parsed with Roslyn → classes, methods, constructors, dependencies extracted
   - Other files are split into text chunks
   - Embeddings are generated by `LocalHashEmbeddingService` (deterministic, no API required)
   - Metadata is stored in SQLite (`SqliteMetadataStore`)
   - Vectors are stored in-memory (`InMemoryVectorSearchService`)

---

## How Incremental Indexing Works

On each incremental run the indexer:

1. Loads all previously indexed files (by repository + branch)
2. Scans current disk state
3. Compares by normalized relative file path and SHA256 hash:

| Case | Action |
|------|--------|
| File on disk, not in index | **Added** — parse, chunk, embed, store |
| File in both, same hash | **Unchanged** — skip |
| File in both, different hash | **Updated** — delete old chunks/vectors, re-index |
| File in index, not on disk | **Deleted** — mark deleted, remove chunks/vectors |

- Deleted files **never** appear in search results or issue recommendations
- Unchanged files are not re-parsed or re-embedded

---

## How Semantic-Style Search Works

Search uses **hybrid scoring**:

```
score = vector_similarity × 0.5
      + keyword_overlap    × 0.3
      + symbol_name_match  × 0.15
      + path_relevance     × 0.05
      × chunk_type_boost
```

- `LocalHashEmbeddingService` generates deterministic 128-dimensional vectors from normalized token hashes
- `InMemoryVectorSearchService` computes cosine similarity
- Results are filtered to exclude deleted files
- Results are ranked by hybrid score

---

## How Issue Analysis Works

Given an issue description:

1. The issue text is passed as a search query
2. Top matching chunks are retrieved using hybrid search
3. Results are grouped by file
4. Each file is scored using average chunk score + path relevance boost
5. Recommendations include:
   - Confidence score
   - Reason why the file is relevant
   - Related symbols (class/method names)
   - Suggested check or action
6. Results are ranked and top N files returned

## How to Run

### Prerequisites

- .NET 9 SDK

### Index the sample repository

```bash
cd src/RepositoryIntelligence.ConsoleApp
dotnet run -- index --path "../../samples/SampleECommerce" --name "SampleECommerce" --branch "main" --mode full
```

### Run incremental index

```bash
dotnet run -- index --path "../../samples/SampleECommerce" --name "SampleECommerce" --branch "main" --mode incremental
```

### Analyze an issue

```bash
dotnet run -- analyze --name "SampleECommerce" --branch "main" --issue "Payment is not triggered after order creation"
```

### Search code

```bash
dotnet run -- search --name "SampleECommerce" --branch "main" --query "order created event service bus payment listener"
```

### Print summary

```bash
dotnet run -- summary --name "SampleECommerce" --branch "main"
```

### Generate Copilot Agent prompt

```bash
dotnet run -- generate-prompt --name "SampleECommerce" --branch "main" --issue "Payment is not triggered after order creation"
```

## How This Helps GitHub Copilot Agent on Large Repositories

On a repository with thousands of files, Copilot Agent has no way to know which 5–10 files are relevant to a bug report. By running Repository Intelligence first:

1. You get a ranked list of files with reasons
2. You can review and curate the list
3. You paste the focused context into a Copilot Agent prompt
4. Copilot Agent works only on the files that matter
5. Changes are less likely to break unrelated functionality

The `generate-prompt` command automates step 3 by producing a ready-to-paste prompt with file list, reasons, and constraints.

- **Pull request creation** — create PRs directly from issue analysis
- **GitHub Actions integration** — run analysis on every new issue automatically
- **Python, Java, TypeScript AST parsing** — extend beyond C# with language-specific parsers
